# Roadmap: Guardião Firmware — Production Hardening

## Overview

Milestone de endurecimento do firmware ESP8266 existente. Ponto de partida: firmware funcional (Protocol v2 completo, OLED UI, Firebase RTDB, OTA). Objetivo: tornar o firmware apto para implantação permanente em campo — sem travamentos por ISR inseguro, sem boot-loops por Firebase instável, sem degradação de heap após dias em operação. Quatro fases em sequência: segurança no loop crítico → resiliência de rede → qualidade de código → cobertura de testes.

## Milestones

- 🚧 **v1.1 Production Hardening** — Fases 1–4 (em andamento)

## Phase Details

### Phase 1: ISR Safety + Boot Resilience

**Goal**: Eliminar as causas de crash mais graves: ISR chamando Serial, device sem tela de erro no boot, e reconexão WiFi inexistente.
**Depends on**: Nothing (ponto de partida)
**Requirements**: REL-01, REL-02, REL-03, OBS-01
**Success Criteria** (what must be TRUE):

1. `ISRWatchDog()` nunca chama `Serial` — apenas sinaliza flag e incrementa contador
2. Quando WiFi cai, dispositivo tenta `WiFi.reconnect()` antes de `ESP.reset()`
3. Se Firebase auth falha repetidamente em `setup()`, OLED exibe tela de erro nomeada antes de reiniciar
4. Qualquer falha crítica de boot (WiFiManager timeout, auth falha definitiva) mostra tela OLED com motivo antes de `ESP.restart()`
   **Plans**: 2 plans

Plans:

- [x] 01-01: Corrigir ISRWatchDog e adicionar WiFi.reconnect() antes de reset
- [x] 01-02: Firebase auth retry com contador máximo e telas OLED de erro de boot

**Status: COMPLETE** — Build SUCCESS, RAM 49.2%, Flash 67.5%

### Phase 2: Firebase Resilience + Observability

**Goal**: Parar o boot-loop agressivo causado por falhas temporárias no Firebase e tornar o status de rede visível ao usuário no OLED.
**Depends on**: Phase 1
**Requirements**: REL-04, REL-05, OBS-02
**Success Criteria** (what must be TRUE):

1. Falha no envio Firebase usa backoff exponencial (mínimo 3 tentativas antes de reset)
2. `getDeviceConfigurations()` é chamado em intervalo próprio (ex: a cada 60 min), não a cada envio de temperatura
3. Desconexão WiFi detectada em `loop()` exibe mensagem de reconexão no OLED em vez de silêncio
   **Plans**: 2 plans

Plans:

- [x] 02-01-PLAN.md — EEPROM pending queue + backoff exponencial Firebase + config fetch timer separado
- [x] 02-02-PLAN.md — drawWifiReconnecting() e WiFi status routing em loop()

**Status: COMPLETE** — Build SUCCESS, RAM 49.5%, Flash 67.5%

### Phase 3: Code Quality

**Goal**: Eliminar as três principais fontes de debt técnico que dificultam manutenção e causam falhas silenciosas em produção.
**Depends on**: Phase 2
**Requirements**: CQ-01, CQ-02, CQ-03
**Success Criteria** (what must be TRUE):

1. `drawOled.h` dividido em módulos focados: `buttons.h`, `alarm.h`, `buzzer.h`, `drawOled.h` (apenas rendering)
2. Nenhuma comparação de float com -127.0 ou 888.0 nos arquivos — substituídas por `bool sensorError`
3. Paths Firebase e strings de tamanho conhecido usam `snprintf` em `char[]` em vez de `String` concatenation
   **Plans**: 2 plans (CQ-02 já implementado na Phase 2)

Plans:

- [x] CQ-02 (sensorError flag) — implementado durante Phase 2 (commit 0c18883)
- [x] 03-01-PLAN.md — Extrair buttons.h, alarm.h, buzzer.h de drawOled.h (CQ-01)
- [x] 03-02-PLAN.md — snprintf para hot-path string building (CQ-03)

**Status: COMPLETE** — Build SUCCESS, RAM 49.6%, Flash 67.6%

### Phase 4: Testing Infrastructure

**Goal**: Estabelecer cobertura de teste para a lógica pura do firmware, garantindo que refatorações futuras não quebrem comportamento crítico silenciosamente.
**Depends on**: Phase 3
**Requirements**: TEST-01, TEST-02
**Success Criteria** (what must be TRUE):

1. `pio test -e native` executa sem erros e com saída de resultado
2. `calculateStatusBitmap()` tem testes que cobrem todos os estados de alarme
3. `checkThresholdAlert()` tem testes para os 4 modos (none, lower, upper, both)
4. Encode/decode EEPROM tem round-trip test (write → read → asserteq)
   **Plans**: 2 plans

Plans:

- [x] 04-01-PLAN.md — Ambiente native + stubs + extração de lógica pura para src/logic.h (TEST-01, TEST-02)
- [x] 04-02-PLAN.md — Testes unitários completos: bitmap, threshold, EEPROM string, pending queue (TEST-02)

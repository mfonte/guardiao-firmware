Pelo bloco abaixo (per D-06, D-07, D-08, D-09, D-10 do CONTEXT.md):

```cpp
else if (!Firebase.ready() && (millis() - sendDataPrevMillis > timerDelay || sendDataPrevMillis == 0))
{
  Serial.println("[WiFi] Firebase not ready. Attempting WiFi reconnect...");
  bool reconnected = false;
  for (int attempt = 0; attempt < 3; attempt++)
  {
    drawBootProgress("Reconnecting...", 0);
    WiFi.reconnect();
    delay(2000);
    if (WiFi.status() == WL_CONNECTED)
    {
      reconnected = true;
      Serial.printf("[WiFi] Reconnected on attempt %d\n", attempt + 1);
      break;
    }
    Serial.printf("[WiFi] Attempt %d failed, status=%d\n", attempt + 1, WiFi.status());
  }

  if (!reconnected)
  {
    Serial.println("[WiFi] All reconnect attempts failed. Resetting.");
    drawResetDevice();
    delay(2000);
    ESP.reset();
  }
  // Se reconectou, Firebase.begin() irá reconectar automaticamente na próxima chamada.
}
```

**Notas de implementação:**

- `WiFi.reconnect()` no ESP8266 é assíncrono — retorna imediatamente, por isso o `delay(2000)` antes de verificar `WiFi.status()`
- `drawBootProgress("Reconnecting...", 0)` reutiliza função existente — progresso fixo em 0 é intencional (não há indicação de progresso real)
- O loop usa `break` para sair cedo se reconectou — não espera as 3 tentativas se a primeira funcionar
- Manter `drawResetDevice()` antes de `ESP.reset()` no fallback (comportamento atual preservado)
- Não usar `Firebase.ready()` dentro do loop de retry — Firebase.ready() verifica token, não WiFi connectivity
